1、课程设计要求
（1）1～4 人/组，以小组为单位， 团队合作完成所选项目；
（2）每小组推选 1 人任组长，组长负责团队合作、项目的进展及安排； 
（3）项目周期： 15- 16 周
（4）成果提交和项目演示时间：第 17 周

2、课程设计内容
以小组为单位,从以下题目中任选一个题目，根据实际设计， 每小组可以合理增减项目内容。
（1）电商用户行为分析系统
项目背景：某电商平台希望深入了解用户购物行为，优化运营策略。提供用户点击、浏览、加购、下单、支付等行为日志数据（含user_id、item_id、category_id、behavior_type、timestamp等字段）。
任务要求：
	使用MapReduce、MySQL或Hive进行数据清洗（过滤异常记录、处理时间格式、去重等）。
	构建分层数据仓库：ODS层（原始日志）、DWD层（清洗后的行为明细，增加日期、小时维度）、DWS层（用户/商品粒度的轻度汇总）。
	设计事实表（如行为事实表）与维度表（用户维度、商品维度、时间维度）。
 实现以下统计指标：
	日活跃用户数（DAU）与周/月活跃用户数。
	商品转化率（从浏览→加购→支付的各环节转化率）。
	用户复购率（购买次数≥2的用户占比）。
	购物车放弃率（加购但未支付的用户比例）。
	将结果导出至MySQL，使用自己熟悉的方式（Spring Boot + ECharts、R语言或者Excel）进行可视化展示（如折线图、漏斗图、柱状图）。
	撰写项目报告，包含需求分析、数仓分层设计、HQL实现、可视化截图及问题总结。
（2）校园一卡通消费数据分析系统
项目背景：某高校后勤部门希望分析学生一卡通消费记录，辅助贫困生认定、优化食堂及商铺运营。提供消费流水数据（含card_id、student_id、consume_time、consume_amount、merchant_type、merchant_name等字段）及学生基本信息表（学院、年级、生源地等）。
任务要求：
	使用MapReduce或Hive进行数据清洗（处理缺失值、异常金额、重复记录，提取消费日期、星期、时段）。
	构建分层数仓：ODS（原始消费流水）、DWD（清洗后明细，增加时段分类如早餐/午餐/晚餐/夜宵）、DWS（学生日消费汇总、商户时段汇总）。
	设计事实表（消费事实表）与维度表（学生维度、商户维度、时间维度）。
 实现以下统计指标：
	客流高峰时段（按小时/星期统计各商户类型的消费次数）。
	学生消费水平分布（月消费总额的分组统计，用于识别低消费群体）。
	贫困生消费特征（结合学生信息，分析低消费群体的消费时段、商户偏好）。
	节假日与平日消费模式对比（如国庆节、寒暑假期间的消费变化）。
	将结果导出至MySQL，使用自己熟悉的方式（Spring Boot + ECharts、R语言或者Excel）展示热力图、箱线图、对比柱状图。
	撰写项目报告，重点说明贫困生消费特征的发现与建议。
（3）短视频平台内容分析系统
项目背景：某短视频平台希望挖掘优质内容与用户行为规律，提升留存与推荐效果。提供用户观看记录（user_id、video_id、watch_duration、video_duration、timestamp、like_flag、comment_flag、share_flag）、视频信息表（video_id、uploader_id、category、publish_time）、创作者信息表。
任务要求：
	使用MapReduce或Hive进行数据清洗（剔除时长异常、重复观看记录，计算完播率字段）。
	构建分层数仓：ODS（原始观看日志）、DWD（清洗后明细，增加完播标记、互动标记）、DWS（用户日/周观看汇总、视频日热度汇总）。
	设计事实表（观看事实表）与维度表（用户维度、视频维度、创作者维度、时间维度）。
 实现以下统计指标：
	视频完播率分布（按类别、按创作者统计平均完播率）。
	用户留存率（次日、7日、30日留存）。
	内容热度排行（基于完播率+点赞+评论+分享的综合热度分，按日/周更新Top N）。
	创作者影响力指数（综合其所有视频的播放量、完播率、互动率，形成指数排名）。
	将结果导出至MySQL或直接生成报表，使用自己熟悉的方式（Spring Boot + ECharts、R语言或者Excel）展示留存曲线、热度排行榜、完播率柱状图。
	撰写项目报告，包含数据清洗逻辑、指标计算公式、Hive优化点（如分区、分桶、MapJoin）及可视化分析结论。
（4）在线教育平台学习行为分析系统
项目背景：某在线教育平台希望了解用户学习习惯与课程质量，为课程优化提供依据。提供用户学习日志（user_id、course_id、chapter_id、video_start_time、video_end_time、watch_duration、video_duration、quiz_score、quiz_max_score、timestamp）、用户注册信息表（user_id、register_date、user_type）、课程信息表（course_id、course_name、category、price、teacher_id）。
任务要求：
	使用MapReduce或Hive进行数据清洗（计算单次学习的实际时长、过滤无效记录、处理测验得分异常值）。
	构建分层数仓：ODS（原始学习日志）、DWD（清洗后明细，增加是否完整学习、章节完课标记）、DWS（课程章节粒度完课率、用户课程粒度学习进度）。
	设计事实表（学习行为事实表、测验事实表）与维度表（用户维度、课程维度、章节维度、时间维度）。
 实现以下统计指标：
	课程完课率（完整看完所有视频或完成所有章节的比例），并区分免费与付费课程。
	知识点掌握度（按章节测验的平均正确率，识别薄弱章节）。
	用户活跃度（日/周学习时长）与期末成绩的关联分析（模拟成绩字段或与测验得分关联）。
	课程流失点分析（计算每章节的跳出率，找到用户最容易放弃的章节）。
	将结果导出至MySQL，使用自己熟悉的方式（Spring Boot + ECharts、R语言或者Excel）展示流失点漏斗图、完课率对比图、活跃度与成绩的散点图。
	撰写项目报告，包含针对流失点的课程改进建议及数仓分层设计图。
（5）物流配送时效分析系统
	项目背景：某物流公司希望通过分析订单配送数据，优化运输路线和时效管理。提供订单数据（order_id、warehouse_id、destination_city、delivery_distance、order_time、dispatch_time、delivery_time、sign_time）、仓库信息表（warehouse_id、warehouse_city）、节假日信息表。
任务要求：
	使用MapReduce或Hive进行数据清洗（处理空值、异常时间顺序、剔除超长配送记录，计算各阶段耗时）。
	构建分层数仓：ODS层（原始订单）、DWD层（清洗明细，增加配送阶段时长：接单时长、运输时长、签收时长）、DWS层（城市/仓库/时段粒度的汇总表）。
	设计事实表（配送事实表）与维度表（仓库维度、城市维度、时间维度、节假日维度）。
实现以下统计指标：
	平均配送时长（按城市、按距离区间、按是否节假日分组）。
	时效达成率（在规定时效内签收的订单比例，可设置不同时效标准）。
	异常配送识别（配送时长超过3倍标准差的订单，并分析其地理位置或时段特征）。
	仓库效率排行榜（基于平均接单时长和运输时效的综合评分）。
	将结果导出至MySQL，使用自己熟悉的方式（Spring Boot + ECharts、R语言或者Excel）展示配送热力图（不同城市颜色深浅表示平均时长）、时效达成率趋势折线图、排行榜柱状图。
	撰写项目报告，包含各阶段耗时分析结论及优化建议（如增加中转仓、调整配送策略）。
（6）银行信用卡欺诈检测分析系统
项目背景：某银行风控部门希望分析信用卡交易流水，识别可疑欺诈模式和用户行为特征。提供交易数据（trans_id、card_id、user_id、trans_amount、trans_time、merchant_category、merchant_location、is_fraud_label（部分有标签或无标签））、用户基本信息表（user_id、age、income_level、credit_score）。
任务要求：
	使用MapReduce或Hive进行数据清洗（过滤测试交易、处理重复交易，提取小时、星期、交易间隔等特征）。
	构建分层数仓：ODS层（原始交易流水）、DWD层（清洗明细，增加小额测试、深夜交易、异常频次等衍生字段）、DWS层（用户日/周交易汇总、商户类别汇总）。
	设计事实表（交易事实表）与维度表（用户维度、商户类别维度、时间维度、地理位置维度）。
实现以下统计指标：
	高频低额交易模式（统计单日内交易次数>5且平均金额<100的用户占比，可能为欺诈测试）。
	深夜交易异常率（23:00-5:00交易占总交易的比例，按用户信用分分组）。
	商户类别欺诈风险排行（若有标签，计算各商户类别的欺诈交易占比；若无标签，可用交易金额异常波动作为代理）。
	用户行为突变分析（与历史7天相比，交易金额或频次增长超过3倍的用户，列出Top可疑用户）。
	将结果导出至MySQL，使用自己熟悉的方式（Spring Boot + ECharts、R语言或者Excel）展示风险评分分布、深夜交易柱状图、Top可疑用户列表的可视化仪表板。
	撰写项目报告，说明每一指标的风控意义，给出规则建议（如触发二次验证的条件）。
（7）社交媒体舆情分析系统
项目背景：某舆情监测公司需要分析社交媒体（如微博、Twitter）上关于特定品牌或事件的海量博文数据，帮助客户了解公众情绪和热点话题。提供博文数据（post_id、user_id、publish_time、content、like_count、retweet_count、reply_count、topic_hashtags）、用户粉丝数等级等辅助信息。
任务要求：
	使用MapReduce或Hive进行数据清洗（过滤非中文或过短博文、处理表情符号，通过简单规则或UDF进行情感极性初步标记（正向/负向/中性））。
	构建分层数仓：ODS层（原始博文）、DWD层（清洗后明细，增加情感标签、发布时间段、是否含图片/视频标记）、DWS层（主题词/小时/用户粉丝等级汇总）。
	设计事实表（博文事实表）与维度表（用户维度、时间维度、话题维度、情感维度）。
实现以下统计指标：
	情感趋势分析（每小时的正面、负面、中性博文数量占比，绘制折线图）。
	热门话题识别（根据hashtag或分词后的高频词，计算词频和互动量加权热度，输出Top10话题）。
	影响力用户排名（综合粉丝数、博文互动量、转发链深度等，产出TOP KOL列表）。
	危机预警指标（负面博文占比突然上升超过阈值，或某话题下负面激增，输出预警时间段和话题）。
	将结果导出至MySQL，使用自己熟悉的方式（Spring Boot + ECharts、R语言或者Excel）展示情感趋势面积图、词云（需额外Python库，Hive侧提供词频表即可）、KOL排行榜条形图。
	撰写项目报告，包含情感分类的UDF实现或规则说明、热门话题计算方法、预警阈值设定依据。
（8）游戏玩家行为分析系统
项目背景：某手游公司希望分析玩家行为数据，提升留存与付费转化。提供玩家登录日志（user_id、login_time、logout_time、online_duration）、游戏内事件流水（user_id、event_type（如level_start/level_complete/purchase/revive）、event_time、level_id、item_id、spend_amount）、玩家注册信息（user_id、register_date、channel、device_type）。
任务要求：
	使用MapReduce或Hive进行数据清洗（处理session分割、重复事件、异常时长，计算每日首次登录、最后登出等）。
	构建分层数仓：ODS层（原始日志）、DWD层（清洗后明细，增加日期、小时、游戏等级段）、DWS层（用户日/周汇总：在线时长、关卡通过数、充值金额）。
	设计事实表（登录事实表、事件事实表）与维度表（用户维度、关卡维度、道具维度、时间维度）。
实现以下统计指标：
	玩家留存率（次日、7日、30日留存，可区分渠道和设备类型）。
	关卡通过率与难度分析（每个关卡首次尝试通过率、平均尝试次数，识别卡点关卡）。
	付费行为分析（付费率、ARPU、ARPPU，付费用户首次付费前的平均在线时长和关卡进度）。
	流失预警模型（基于最后一次登录时间、近期在线时长下降、关卡停滞等特征，定义流失风险用户等级）。
	将结果导出至MySQL，使用自己熟悉的方式（Spring Boot + ECharts、R语言或者Excel）展示留存曲线、关卡通过率漏斗图、付费用户特征对比雷达图。
	撰写项目报告，包含流失预警规则的设计思路及基于Hive SQL的具体实现，提出游戏优化建议（如调整卡点关卡难度、推送优惠券）。
提交内容
	小组提交内容
一、文档类（占30%）
1. 项目报告（PDF格式）
需求分析：项目背景、业务目标、指标定义（例如：什么是“用户复购率”？如何计算“完播率”？）。
系统设计：
•	数仓分层设计图（ODS → DWD → DWS → ADS/结果层），说明各层作用。
•	事实表与维度表设计（表名、字段、分区键、存储格式、压缩方式）。
•	数据流向图（源数据 → 清洗 → 分层建模 → 指标计算 → 导出可视化）。
实现过程：
•	数据清洗逻辑（MapReduce或HiveQL脚本片段，并解释处理了哪些脏数据）。
•	核心HiveQL代码（每个统计指标给出关键SQL，可附部分截图）。
•	导出与可视化方案（Sqoop命令、可视化工具/框架搭建步骤）。
测试与验证：抽样对比原始数据与计算结果，说明结果的合理性。
项目总结：遇到的问题及解决方案、小组分工、未完成功能、改进方向
附录：小组人员名单、学号、任务分配表
2. 小组分工与贡献说明（可放在项目报告内）
•	列出每位成员完成的具体任务（如：数据清洗、建表、指标A实现、可视化、文档撰写）。
二、代码与脚本类（占40%）
3. 数据清洗脚本
•	若使用MapReduce：提供Java源码及打包后的jar包。
•	若使用Hive：提供清洗用的HiveQL脚本（.hql或.sql文件），包含INSERT OVERWRITE TABLE ... SELECT ... 等转换逻辑。
4. 数仓建表语句
•	一个完整的SQL脚本文件（create_tables.hql），顺序执行可创建ODS、DWD、DWS及结果表，并包含必要的表注释、分区定义、存储格式（如STORED AS ORC）和压缩配置。
5. 指标计算HiveQL脚本
•	每个指标一个单独的脚本文件（如kpi1_dau.hql, kpi2_conversion_rate.hql），方便教师独立运行验证。
•	脚本中应包含必要的参数设置（如set hive.exec.dynamic.partition=true;）。
6. 数据导出脚本
•	Sqoop导出命令的脚本文件（export_to_mysql.sh），包含连接信息、表映射、增量或全量导出逻辑。
7. 可视化项目源码
•	如使用Spring Boot + ECharts：提供完整项目源码（可打包为.zip），包含后端接口、前端页面。
•	若使用其他工具（如Tableau、Power BI）：提供打包的工作簿文件及数据源连接说明。
三、数据与结果类（占15%）
8. 样例数据及处理后的数据文件（可选，若数据量不大）
•	原始数据的前1000行（.txt或.csv）。
•	清洗后DWD层表的导出样例（前100行）。
•	最终指标结果表的数据（CSV格式，至少包含Top 10或全量结果）。
9. 可视化截图或录屏
•	仪表板完整截图（不少于3张），展示所有要求的图表。
•	如有动态交互效果，可提供1-2分钟的录屏（.mp4），或在线访问链接（若部署在服务器）。
四、演示与答辩类（占15%）
10. 答辩PPT
•	10-15页，内容包括：项目背景、技术架构、核心难点、指标展示、小组分工、心得体会。
•	课堂上进行5-8分钟的演示和答辩（教师可随机提问）。
